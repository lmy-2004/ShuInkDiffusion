# 进 Latent 前风格化与深度/描边条件解耦：调研与实施说明

本文说明在实时流式管线中，为何将 **深度图 / SoftEdge 图生成与编码** 与 **主图像 VAE 编码** 拆开，如何用 **shader 式可调前处理** 得到 `stylized_x` 再送入 `encode_image`，以及 **`stylization_kwargs` 从 YAML 到 Demo 的透传** 与 **stylized 预览缓存** 的落地方式。实现主文件为 [`Live2Diff/live2diff/pipeline_stream_animation_depth.py`](../../Live2Diff/live2diff/pipeline_stream_animation_depth.py)（非入口 `demo/main.py`）。

---

## 一、调研了哪些资料与方向

调研不追求复现某一篇论文的完整方法，而是对齐三类已有共识，并落到**可实时、可解释、可调参**的工程形态。

### 1. 单目深度作为结构先验（与本仓库深度分支一致）

- **脉络**：从 MiDaS、**DPT**（Dense Prediction Transformer）等单目深度估计工作开始，单帧 RGB → 相对深度图已成为视频/图像条件扩散里的常见几何先验；本工程已使用 DPT 类权重（如 `dpt_hybrid_384`）产生 `depth_map` 再经 VAE 得到 `depth_latent`。
- **结论取向**：深度适合表达**前后景层次与空间布局**，适合作为 **UNet 侧结构条件**；若把「风格化后的 RGB」再喂回深度估计，会引入**外观漂移**，使条件与真实几何脱节（尤其在强风格、强雾化时）。

### 2. 多条件控制扩散（ControlNet / 空间条件注入）

- **脉络**：在扩散模型中并行注入多种空间条件（边缘、深度、语义等）已是标准做法；Live2Diff 流式 UNet 侧已同时支持 `depth_sample` 与 `softedge_sample` 等通路。
- **结论取向**：**不同条件应对齐同一「物理/几何」语义**（原图上的深度与边缘），而**外观风格**更适合在 **VAE 输入域** 或 **像素前处理域** 完成，再进入主 latent，避免「条件图本身被风格污染」导致控制信号语义模糊。

### 3. 非真实感渲染（NPR）与图形学中的深度驱动效果

- **脉络**：**大气透视（atmospheric perspective）**、基于深度的雾效、远近虚实，在经典图形学与实时渲染（shader）中普遍用 **lerp**、**smoothstep**、**模糊/幂次** 等可组合算子表达；边缘增强则常与 **Sobel / 梯度幅值**、**线检测**（本仓库 SoftEdge 含 classical 与 PidiNet 路径）同源。
- **结论取向**：水墨/雾化类视觉目标与「深度调制雾白混合 + 边缘压暗」一类 **image-based stylization** 高度同构；用 **Torch 端可微、可批处理** 的同样算子，便于与现有 GPU 推理同设备执行，且参数可暴露给 Demo 调参。

### 4. 工程与产品向：warmup 与在线推理分布一致

- **问题**：若仅在 `__call__` 做风格化而 `prepare()` 仍用原图编码，会导致 **warmup 缓冲与正式帧 latent 分布不一致**，表现为启动后短暂风格跳变或不稳定。
- **结论**：**同一套前处理** 必须同时挂在 `prepare()` 与 `__call__()` 两条路径上（本实现已统一）。

## 二、调研结论（设计决策）

1. **先基于 `base_x`（原预处理 RGB）生成 `depth_map` 与 `softedge_map`，并各自编码为 `depth_latent` / `softedge_latent`**  
   保证结构条件反映**原始输入几何**，不被风格化后的对比度、色调、雾化所扭曲。

2. **仅将 `stylized_x` 送入 `encode_image()` 得到主路 `x_t_latent`**  
   让扩散主分支「看到」的是**期望的水墨/雾化外观**，与 LoRA、prompt 意图一致。

3. **风格化采用可解释的 shader 式管线**  
   - **降饱和**：Rec. 601 亮度与 `saturation_scale` 的 lerp。  
   - **深度支路**：高斯模糊、`pow`、**smoothstep** 得到平滑蒙版 `pro_depth`。  
   - **雾化**：模糊原图后与白、与原图的 **lerp**（`fog_white_mix`、深度调制）。  
   - **描边支路**：对 softedge 做模糊、`pow`、smoothstep 得 `pro_outline`，再与雾图 **lerp** 向暗部收边。  
   算子与实时 NPR 文献/教材中的常见表述一致，便于迭代默认值与 UI 说明。

4. **拆分 API：`build_*_map` 与 `encode_*_map` 分离**  
   避免在同一帧内对「已风格化图像」重复算 depth/edge，也避免逻辑上误用风格图重建条件（见 `__call__` 中 softedge 在风格化前已构建；`encode_softedge_map` 始终针对该 `softedge_map`）。

5. **可选预览与调试**  
   缓存「进 latent 前」的 stylized 张量，便于与最终输出对比调参；必要时可扩展 pro_depth / pro_outline 的独立导出（当前管线已写入 `last_processed_depth_map` / `last_processed_outline_map` 供后续接 API）。

## 三、如何在本项目中落地

### 1. 数据流（单帧）

1. `x = preprocess(输入)` → 记为语义上的 **base_x**。  
2. `depth_map = build_depth_map(x)`，`depth_latent = encode_depth_map(depth_map)`。  
3. 若需要 softedge（`use_softedge` / `softedge_debug` / `enable_stylize_preprocess`）：  
   `softedge_map = build_softedge_map(x)`（仍基于 **base_x**）。  
4. 若 `enable_stylize_preprocess` 且 `softedge_map` 非空：  
   `x = build_stylized_image(x, depth_map, softedge_map)` → **stylized_x**。  
5. `x_t_latent = encode_image(x)`（此处 `x` 已为 stylized）。  
6. `softedge_latent`：在开启 softedge 时用 **步骤 3 的 `softedge_map`** 做 `encode_softedge_map`，**不**用风格化后的 `x` 重算（除非关闭 stylize 且逻辑要求重建的少数分支；当前主路径已固定用首次 map）。

### 2. 核心实现文件

| 模块 | 路径 | 说明 |
|------|------|------|
| 流式管线 | [`Live2Diff/live2diff/pipeline_stream_animation_depth.py`](../../Live2Diff/live2diff/pipeline_stream_animation_depth.py) | `build_depth_map` / `encode_depth_map` / `build_softedge_map` / `encode_softedge_map`；`_smoothstep`、`_lerp`、`_gaussian_blur`、`_desaturate`、`build_stylized_image`；`prepare()` 与 `__call__()` 共用上述顺序；`set_stylization` 更新超参 |
| 封装 | [`Live2Diff/live2diff/utils/wrapper.py`](../../Live2Diff/live2diff/utils/wrapper.py) | `set_stylization`、`get_last_stylized_image()`；`get_debug_info` 中带 stylization 状态 |
| Demo 参数模型与装配 | [`Live2Diff/demo/vid2vid.py`](../../Live2Diff/demo/vid2vid.py) | Pydantic 字段与 `stylization_kwargs` 从 cfg 读取、`stream.set_stylization` |
| 默认 / 演示配置 | [`Live2Diff/configs/base_config.yaml`](../../Live2Diff/configs/base_config.yaml)、[`Live2Diff/demo/demo_cfg.yaml`](../../Live2Diff/demo/demo_cfg.yaml) | `stylization_kwargs` 块 |
| 预览缓存与 HTTP | [`Live2Diff/demo/app.py`](../../Live2Diff/demo/app.py)、[`Live2Diff/demo/connection_manager.py`](../../Live2Diff/demo/connection_manager.py) | 会话中缓存 stylized 字节；`GET /api/stylized/{user_id}` |

### 3. 可配置参数（`stylization_kwargs`）

| 键 | 含义（概要） |
|----|----------------|
| `enable_stylize_preprocess` | 是否启用整条 stylize + 用 stylized 进 `encode_image` |
| `depth_blur_sigma` / `depth_power` | 深度蒙版平滑与对比度（幂次） |
| `depth_smoothstep_min` / `depth_smoothstep_max` | 深度 smoothstep 区间 |
| `saturation_scale` | 降饱和强度（与灰度 lerp） |
| `image_blur_sigma` | 雾化支路对原图的模糊 |
| `fog_white_mix` | 雾底：模糊图与白色的混合 |
| `outline_blur_sigma` / `outline_power` | 描边蒙版平滑与对比度 |
| `outline_smoothstep_min` / `outline_smoothstep_max` | 描边 smoothstep 区间 |

具体默认值以 `base_config.yaml` 为准；`demo_cfg.yaml` 中可覆盖为更偏水墨/雾化的试验值。

### 4. 预览接口

- 推理路径在 `build_stylized_image` 内更新 `last_stylized_map`（及处理后的 depth/outline 缓存）。  
- Wrapper 通过 `get_last_stylized_image()` 转为 PIL。  
- `app.py` 将会话内 **stylized 预览** 暴露为 **`/api/stylized/{user_id}`**，便于与最终帧对照调参。

## 四、验证与局限

- **已做**：对改动相关 Python 文件执行 `py_compile`；静态检查（lints）无报错。  
- **未做**：完整拉流 Demo 下的主观画质与参数手感需现场联调。  
- **局限**：stylize 与 softedge 在开启时增加每帧算子开销；极强 blur/sigma 可能拖慢小步推理预算；PidiNet 路径为 CPU/辅助库依赖，与纯 classical 路径行为不同，需在 `softedge_mode` 与 stylize 组合下分别调参。
