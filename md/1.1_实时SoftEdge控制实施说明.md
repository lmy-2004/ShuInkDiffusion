# 实时 SoftEdge 控制实施说明

## 目标

本次实现的目标不是把项目直接改造成完整 `ControlNet SoftEdge`，而是在当前 `Live2Diff` 的实时链路上，增加一条和 `depth` 并行的 `softedge` 条件控制分支，并保持：

- warmup 和 streaming 行为一致
- `none/xformers/tensorrt` 三条路径接口一致
- Demo 可以直接开关、调参和预览 softedge

## 架构取舍

### 为什么没有直接复刻完整 ControlNet

标准 `ControlNet Depth/SoftEdge` 的控制方式是：

- 输入 `controlnet_cond`
- 额外侧路网络逐层产生 `down_block_additional_residuals`
- 在 UNet 多尺度位置注入 residual

这条路控制力更强，但对当前项目第一阶段并不合适，原因很直接：

1. 每个 denoising step 都要多跑一条控制侧路，实时性压力太大。
2. 当前工程已经深度绑定 warmup UNet、streaming UNet 和 TensorRT UNet 输入签名，完整 ControlNet 改动面过大。
3. 当前项目更适合先复用已有的 `depth -> latent -> cond mapping -> add` 轻量路径。

### 这次实际采用的方案

当前实现采用的是 `early-fusion softedge`：

1. 输入帧先在扩散循环外提取 softedge。
2. softedge 图复用现有 VAE encoder 编成 latent。
3. 在 UNet 输入早期通过新增的 `edge_flow_conv_in` 做映射。
4. `depth` 与 `softedge` 在 `conv_in` 后直接与主特征相加。

这条方案的优点是：

- 只在每帧增加一次 softedge 提取和一次 VAE 编码
- 不需要每个 timestep 额外跑控制网络
- 更容易接到现有 TensorRT 导出链

## 已实现内容

### 1. SoftEdge 主链

已经在 `Live2Diff/live2diff/pipeline_stream_animation_depth.py` 中加入：

- `set_conditioning()`
- `initialize_softedge_branch()`
- `encode_softedge()`
- classical softedge 提取
- 可选 `pidinet` 模式入口
- `softedge_latent_buffer`

当前默认策略：

- `softedge_mode=classical` 时，使用 `gray -> blur -> sobel -> blur -> normalize` 的轻量 softedge
- `softedge_mode=pidinet` 时，会尝试加载 `controlnet_aux` 的 `PidiNetDetector`
- 如果 `pidinet` 不可用，会自动回退到 `classical`

### 2. UNet 条件注入

已经在以下两个文件中新增 `edge_flow_conv_in`：

- `Live2Diff/live2diff/animatediff/models/unet_depth_streaming.py`
- `Live2Diff/live2diff/animatediff/models/unet_depth_warmup.py`

并把 forward 扩成：

- `depth_sample`
- `softedge_sample`

两个条件都在 `conv_in` 后注入。

### 3. 无重训可用初始化

新分支如果直接随机初始化，第一版几乎不会产生稳定控制效果。为了解决这个问题，这次实现里增加了：

- `init_softedge_from_depth()`

它会把 `depth` 的 `flow_conv_in` 权重复制给 `edge_flow_conv_in`，让新分支从一个可工作的结构映射开始，而不是从纯随机状态开始。

### 4. TensorRT 输入适配

已经同步扩展：

- `Live2Diff/live2diff/acceleration/tensorrt/models.py`
- `Live2Diff/live2diff/acceleration/tensorrt/engine.py`
- `Live2Diff/live2diff/utils/wrapper.py`

关键变化：

- UNet TensorRT 输入新增 `softedge_sample`
- engine feed dict 同步新增 `softedge_sample`
- engine prefix 增加 `cond-v2`，避免复用旧版不兼容引擎

### 5. Demo 与调试

已经在 Demo 中加入下面的运行时参数：

- `use_softedge`
- `softedge_scale`
- `depth_scale`
- `softedge_mode`
- `softedge_debug`

并新增 softedge 调试链路：

- `StreamAnimateDiffusionDepthWrapper.get_last_softedge_image()`
- `/api/softedge/{user_id}`
- debug 信息中的 `has_softedge`、`softedge_scale`、`softedge_mode`、`softedge_backend`

## 验证方式

### 功能验证

至少确认下面几点：

1. warmup 阶段不报错，说明 warmup UNet 已接通 `softedge_sample`
2. streaming 阶段不报错，说明主推理链已接通 `softedge_sample`
3. `use_softedge=false` 时输出可退化为 depth-only
4. `/api/softedge/{user_id}` 能返回 softedge 调试图
5. debug 信息能看到 `softedge_backend`

### 实时性验证

建议重点看：

1. `inference_time`
2. `depth_time`
3. 开启 softedge 前后的整体 FPS
4. `classical` 与 `pidinet` 的单帧耗时差异

### 视觉验证

建议至少对比三组：

1. `depth-only`
2. `depth + classical softedge`
3. `depth + pidinet softedge`

重点观察：

- 人物边缘是否更稳
- 建筑轮廓是否更干净
- 水墨留白边界是否更清晰
- 快速运动时边缘是否比 depth-only 更少漂移

## 后续建议

如果第一版 `early-fusion softedge` 已经明显改善边界稳定性，下一步最值得做的是：

1. 针对 `softedge_scale` 做运动感知调度
2. 为 softedge 加入更稳定的时序平滑
3. 评估是否需要进入 `ControlNet-lite residual` 路线
