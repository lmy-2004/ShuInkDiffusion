# softedge / depth 参数趋势与稳定性分析报告

## 目标

本文专门分析当前 benchmark 热力图里两个最直观的问题：

- 为什么 `softedge_scale` 增大后，稳定性差值反而整体变差。
- 为什么 `depth_scale` 的控制效果看起来不够理想，改善有限且不稳定。

本文不重复介绍整个项目，而是把以下三部分串起来：

1. 当前热力图到底在量什么。
2. 当前代码里 `softedge` / `depth` 是怎样进入生成链路的。
3. 外部研究与工程经验如何解释这类趋势。

---

## 一、先给结论

如果只看这张图，本次最重要的结论有四条：

1. `softedge_scale` 增大后稳定性整体变差，这不是偶然噪声，而是当前实现下的主趋势。
2. `depth_scale` 不是完全无效，但它更像“低频结构锚点”，只能在 `softedge` 较低时提供有限补偿，无法压住高频边缘抖动。
3. 当前 `softedge` 分支更接近“轮廓增强器”，还不是“时序稳定器”。
4. 图里的局部起伏不应全部按真实参数机理解读，因为 benchmark 过程本身还混入了时序状态残留和单次读数噪声。

如果把图上的趋势压缩成一句话，就是：

> 当前系统里，`softedge` 放大的是高频轮廓条件，而高频轮廓恰好最容易在逐帧归一化、逐帧提取、逐帧注入的链路中变成闪烁源；`depth` 只能部分稳定大结构，但它自身也存在逐帧尺度漂移，因此无法根治这种问题。

---

## 二、这张图实际测量的是什么

这张热力图的横轴是 `softedge_scale`，纵轴是 `depth_scale`，颜色和格子里的数字来自 benchmark 导出的 `score`。

但这个 `score` 不是“绝对稳定度”，而是：

- `temporal_stability_delta_avg_30s`
- 即最近 30 秒内 `重绘稳定度 - 原画稳定度` 的滑动均值

当前前端 benchmark 面板就是这样读取分数的：

- `BenchmarkPanel.svelte` 中读取 `debugInfo.temporal_stability_delta_avg_30s`
- benchmark json 的 `meta.score_field` 也被写死为 `temporal_stability_delta_avg_30s`

这意味着：

- 数值越大越好
- 数值为负，表示模型输出比原视频更不稳定
- 数值越负，表示模型额外引入的时序抖动越强

因此图中所有格子都小于 0，不是在说“系统完全不可用”，而是在说：

> 对同一段视频，在这些参数组合下，模型输出相对原视频都额外引入了不同程度的抖动。

从图面可以直接读出两个总体趋势：

- 当 `softedge_scale` 从 `0` 增加到 `1` 时，分数大多从约 `-1.6 ~ -2.4` 恶化到 `-3.3 ~ -3.9`
- `depth_scale` 在 `softedge_scale` 很低时偶尔有改善，但一旦 `softedge_scale` 高起来，`depth` 的改善就明显不够了

也就是说，图里最重要的趋势是：

- **横向主趋势很强**：`softedge` 越大越差
- **纵向趋势较弱**：`depth` 只能局部缓解

## 三、为什么 `softedge_scale` 增大后反而更差

这部分是整张图最核心的问题。

### 1. `softedge` 本质上是高频条件，而高频最容易抖

当前实现里的 `softedge` 不是语义分割，也不是实例级结构图，而是通过经典边缘提取生成：

- 先转灰度
- 再高斯模糊
- 再做 Sobel 梯度
- 再对梯度幅值归一化
- 最后编码进 latent

这类信号本身就是高频信号，天然对以下因素敏感：

- 亮度轻微变化
- 局部对比度变化
- 头发、脸部边缘、手指轮廓的小位移
- 压缩噪声、运动模糊、采样误差

而高频条件一旦不稳定，扩散模型不会把它自动“平均掉”，反而常常会把它画得更明显。

对水墨风格尤其如此，因为水墨最敏感的正是：

- 轮廓是否稳定
- 边界是否干净
- 墨线粗细是否跳动
- 留白边缘是否忽灰忽白

所以 `softedge` 越强，并不等于“越稳”，在当前实现下更可能是“越把边缘抖动送进模型”。

### 2. `softedge` 是逐帧归一化的，强度标尺每帧都在变

当前 `softedge` 图在构造时会经过 `_normalize_condition_map()`，它是按每一帧自己的 `min/max` 归一化：

```python
def _normalize_condition_map(self, cond_map):
    cond_min = cond_map.amin(dim=(-2, -1), keepdim=True)
    cond_max = cond_map.amax(dim=(-2, -1), keepdim=True)
    cond_map = (cond_map - cond_min) / (cond_max - cond_min).clamp_min(1e-6)
    return cond_map
```

这个问题的关键不在“归一化本身错了”，而在“它完全是逐帧的”。

结果是：

- 某一帧轮廓稍弱，就会被重新拉满
- 某一帧局部边缘稍强，就会改变整帧对比度标尺
- 相邻帧即使内容几乎一致，`softedge` 的整体动态范围也可能不同

于是模型收到的并不是“同一条边的稳定描述”，而是：

> 每一帧都重新拉伸过强度、重新定义过边缘对比度的条件图。

当 `softedge_scale` 很小时，这种问题影响有限。  
当 `softedge_scale` 增大时，这种逐帧标尺抖动会被直接放大。

### 3. `softedge` 是在 UNet 入口直接相加注入，不是温和控制

当前流式 UNet 的注入方式非常直接：

```python
sample = self.conv_in(sample)
if depth_sample is not None:
    depth_sample = self.flow_conv_in(depth_sample)
    sample = depth_sample + sample
if softedge_sample is not None:
    softedge_sample = self.edge_flow_conv_in(softedge_sample)
    sample = softedge_sample + sample
```

这里有两个重要含义：

1. `softedge` 不是后处理，而是直接参与主干特征。
2. `softedge_scale` 不是“小提示”，而是在入口处线性放大条件分支。

也就是说，当 `softedge_scale` 提高时，模型更强地被边缘条件牵引。  
如果边缘条件本身时序稳定，那会有帮助。  
但如果边缘条件本身就带着逐帧波动，那么 `softedge_scale` 提高后，放大的就是波动本身。

### 4. `softedge` 分支没有经过专门训练，映射层只是从 `depth` 拷贝初始化

这是当前代码里一个非常关键、但很容易被忽略的点。

`edge_flow_conv_in` 的初始权重不是专门针对边缘分布训练出来的，而是直接从 `depth` 的映射层复制过来的：

```python
def init_softedge_from_depth(self):
    if hasattr(self, "flow_conv_in") and hasattr(self, "edge_flow_conv_in"):
        self.edge_flow_conv_in.load_state_dict(self.flow_conv_in.state_dict())
```

这在工程上是合理的冷启动办法，但它有明显副作用：

- `depth` 分支面对的是相对低频、连续、结构性的信号
- `softedge` 分支面对的是高频、稀疏、尖锐、动态范围不稳定的信号

这两类输入的统计分布并不一样。

所以当前 `softedge` 分支更像是：

- 有一条通路
- 能起作用
- 但不是为边缘时序稳定专门适配过的通路

这会导致一个非常典型的现象：

- `softedge_scale` 较小时，它能略微增强轮廓
- `softedge_scale` 较大时，分支的输入分布偏差、逐帧归一化抖动、高频敏感性会一起暴露出来

于是热力图就出现了“edge 越大越差”的主趋势。

### 5. 当前没有对 `softedge` 做时序平滑，也没有运动感知调度

文档里其实已经提前指出了这件事：`softedge` 下一步最值得做的是

- 运动感知调度
- 更稳定的时序平滑

但当前推理逻辑里，`softedge_scale` 仍然是静态数值，并没有随着运动强度变化。

这意味着：

- 静止场景下，本该更保守
- 现在却仍然按固定强度把轮廓条件灌进模型

而静止场景恰恰最容易暴露“轮廓细闪”。  
所以在低运动视频里，强 `softedge` 往往不是加分项，反而是放大闪烁的加速器。

## 四、为什么 `depth_scale` 的效果看起来不够理想

### 1. `depth` 更偏低频结构，天然只能稳“大形”，不擅长稳“细边”

`depth` 的长处是：

- 前后层次
- 主体与背景的大结构关系
- 粗粒度构图约束

它的短板是：

- 不能精确表达轮廓笔锋
- 不能稳定细线粗细
- 不能直接表达水墨留白边界

所以 `depth` 对稳定性的帮助本来就更偏“防大结构漂移”，而不是“压住局部细闪”。

这和 `softedge` 是互补关系，而不是替代关系。

问题在于当前图里真正主导负分的，很可能不是大结构漂移，而是：

- 边缘毛刺闪烁
- 线条粗细跳动
- 纹理和留白边缘的小尺度抖动

这些恰好不是 `depth` 最擅长控制的。

因此图里会出现一种视觉感受：

- `depth` 好像“有点用”
- 但远没有预期里那么有用

这并不奇怪，因为它控制的不是当前最主要的误差源。

### 2. `depth` 自己也有逐帧归一化漂移

当前 `depth_map` 的构造同样是逐帧 `min/max` 归一化：

```python
depth_min = depth_map.amin(dim=(-2, -1), keepdim=True)
depth_max = depth_map.amax(dim=(-2, -1), keepdim=True)
depth_map_norm = (depth_map - depth_min) / (depth_max - depth_min).clamp_min(1e-6)
```

这意味着：

- 相邻帧深度整体范围稍有变化
- 或检测器某些局部值有轻微波动
- 都会改变整帧深度图的缩放关系

这会导致 `depth_scale` 虽然在增强结构约束，但增强的是一个自身也在轻微波动的条件。

于是它的作用会呈现出：

- 有帮助，但不稳定
- 在某些格子里改善明显
- 在另一些格子里改善不明显，甚至回退

这正是热力图纵向趋势不够整齐的一个重要原因。

### 3. 单帧深度先验本身不等于时序稳定深度

当前项目用的是单帧深度估计再进入视频扩散链路。  
这类方案通常能提升结构一致性，但不等于获得了“视频级时序稳定深度”。

外部研究也支持这一点。  
[ChronoDepth](https://arxiv.org/html/2406.01493v1) 在摘要和引言里都明确强调：

- 视频深度的难点不仅是单帧精度
- 更重要的是跨帧一致性
- 单帧深度方法容易因为尺度歧义产生 flicker

这和当前项目的观察非常吻合：  
`depth` 是 useful prior，但它本身并不是时序稳定模块。

### 4. `depth` 只能补结构，补不了每帧新增噪声

当前推理链里，图像 latent 仍然在逐帧加噪：

```python
img_latent = retrieve_latents(self.vae.encode(image_tensors), self.generator)
noise = self._sample_noise_like(img_latent, self.prev_noise_latent)
self.prev_noise_latent = noise.detach().clone()
x_t_latent = self.add_noise(img_latent, noise, 0)
```

虽然已经有 noise control，但当前默认配置非常保守：

- `min_noise_rate: 0.95`
- `max_noise_rate: 1.0`

这几乎意味着：即使低运动场景，也仍然保留了绝大部分 fresh noise。

于是 `depth` 面对的局面其实是：

- 一边努力稳定大结构
- 另一边系统还在持续引入新的随机纹理扰动

在这种前提下，`depth` 的改善空间天然就被压缩了。

也就是说，`depth` 不是没起作用，而是它在和更强的扰动源对冲。

## 五、为什么图里不是完全单调，而是有一些局部异常格子

这部分不能忽略，因为它直接关系到“这张图能解释到什么程度”。

### 1. benchmark 每个组合只读一次分数，不是多次平均

当前 benchmark 对每个参数组合的流程是：

1. 跳回视频开头
2. 改参数
3. 等待一段时间
4. 只读取一次当前 `delta_avg_30s`

这意味着每个格子的 `score` 带有明显单次观测噪声。

### 2. 分数本身还是 30 秒滑动均值

也就是说，一个格子的得分并不只由当前这一小段帧决定，还受过去窗口内的历史样本影响。

于是图里的局部不规则，例如：

- 某个中间格子突然略好一点
- 某一列里有单个格子反弹

并不一定代表真实机理发生了结构性转折。

### 3. benchmark 切换组合时没有显式调用后端 reset

这是当前实验逻辑里非常重要的一条。

前端 `prepareCombo()` 里做的是：

- `jumpToStart()`
- 更新 `pipelineValues`

但没有调用后端 `/api/reset/{user_id}`。

而服务端其实是提供了这个 reset 接口的，它会：

- 清空输出队列
- 清空缓存
- 调用 `pipeline.reset_runtime_state()`

如果 benchmark 不显式 reset，那么不同参数组合之间可能共享一部分：

- warmup 状态
- 上一帧输入/输出缓存
- temporal delta 滑动窗
- 噪声复用状态

这会让热力图出现“组合之间互相污染”的风险。

因此这张图的解读应当分两层：

- **大趋势可信**：`softedge` 增大整体更差，这个趋势很强，不太可能只是噪声
- **局部格子不宜过读**：它们可能混有滑动窗残留、未 reset 的历史状态和单次读数波动

## 六、外部研究如何解释当前现象

这里不做泛泛综述，只挑和当前图最相关的结论。

### 1. 多帧同步视频扩散：单帧各自去噪，天然难达成稳定共识

[Highly Detailed and Temporal Consistent Video Stylization via Synchronized Multi-Frame Diffusion](https://arxiv.org/abs/2311.14343) 明确指出：

- 现有图像扩散模型扩到视频时，难以同时兼顾细节和时序一致性
- 关键原因之一是各帧没有从去噪早期就共享信息
- 他们的解决方向是让多帧同步去噪，并在早期就共享结构和颜色分布信息

这和当前项目的情况高度对应：

- 现在 `softedge` / `depth` 都是逐帧提取、逐帧归一化、逐帧注入
- 并没有形成真正的“跨帧条件共识”

所以当 `softedge` 变强时，它更像是在放大每帧自己的边缘判断，而不是把边缘稳定下来。

### 2. 实时视频风格化：一致性不是越强越好，但必须可控

[Interactive Control over Temporal Consistency while Stylizing Video Streams](https://arxiv.org/abs/2301.00750) 提醒了一个很重要的点：

- 逐帧风格化天然会闪
- 风格化任务不是要把所有 flicker 完全消灭
- 关键是让一致性变成可控量，而不是固定地强推某个条件

这恰好解释了为什么当前静态 `softedge_scale` 不理想：

- 它没有区分高运动和低运动
- 没有区分“该保守稳定”还是“可以保留一些笔触变化”

所以 `softedge` 一旦拉高，就可能同时放大了：

- 有价值的轮廓
- 没价值的抖动

### 3. 水墨视频研究：真正敏感的是留白、轮廓和主体尺度，不是只有 depth

[Instance-Aware Coherent Video Style Transfer for Chinese Ink Wash Painting](https://www.ijcai.org/proceedings/2021/114) 给出的结论与当前问题非常贴近：

- 中国水墨视频的难点不是简单黑白化
- 而是保持 temporal coherence，同时保留 freehand brushstroke
- 他们专门强调了：
  - multi-frame fusion
  - instance-aware style transfer
  - background white space
  - contour depiction

这直接支持当前图里的现象判断：

- `depth` 只能提供部分大结构约束
- 但水墨真正敏感的是留白边界和轮廓笔触
- 如果边缘条件没有时序稳定化，就会比普通风格更容易暴露抖动

### 4. 视频深度研究：深度的一致性本身就是独立难题

[Learning Temporally Consistent Video Depth from Video Diffusion Priors](https://arxiv.org/html/2406.01493v1) 明确指出：

- 视频深度的核心目标之一就是消除由单帧尺度歧义带来的 flickering
- 单帧深度精度提升，并不自动等于跨帧一致性提升

所以在当前项目中，把 `depth_scale` 调大后没有获得理想中的稳定提升，是符合外部研究经验的。

根本原因不是“depth 没价值”，而是：

- 单帧 depth 先验本身还不够时序稳定
- 再叠加逐帧归一化和 fresh noise 注入，改善空间会被进一步压缩

## 七、把代码逻辑和热力图趋势一一对应

这里给出一个更直接的因果链。

### 现象 A：`softedge_scale` 增大，分数整体变差

对应逻辑链：

1. `softedge` 是高频边缘图
2. 边缘图逐帧 `min/max` 归一化
3. 相邻帧边缘强度标尺会漂移
4. `softedge` 通过 `edge_flow_conv_in` 直接加到 UNet 入口特征
5. `softedge_scale` 越大，这种漂移被放大得越厉害
6. 光流 warp error 对边缘粗细与位置波动很敏感
7. 因而 `e_out` 变大，`delta_avg_30s` 更负

### 现象 B：`depth_scale` 只能局部改善

对应逻辑链：

1. `depth` 主要稳定的是主体层次和大结构
2. 当前主要负分源更可能是边缘细闪而不是大结构崩坏
3. `depth` 自己也有逐帧归一化漂移
4. 系统还在逐帧加入 fresh noise
5. 所以 `depth` 只能在低 `softedge` 区域提供有限改善
6. 当 `softedge` 已经很强时，`depth` 的补偿就不够了

### 现象 C：个别格子不完全单调

对应逻辑链：

1. benchmark 每个组合只读一次滑动窗分数
2. 参数切换时没有显式 reset 后端状态
3. warmup / noise / delta window 可能跨组合残留
4. 因此局部格子会混入实验过程噪声

## 八、最终判断

结合代码、指标和外部资料，当前这张图最合理的解释是：

### 1. `softedge` 变强后更差，主要不是因为“edge 没意义”

而是因为当前实现下它满足了最容易诱发闪烁的几个条件：

- 高频
- 逐帧提取
- 逐帧归一化
- 直接入口注入
- 分支未针对 edge 分布专门训练
- 没有时序平滑
- 没有运动感知调度

所以它现在更像“能让轮廓更显眼”，但还不是“能让轮廓更稳定”。

### 2. `depth` 看起来不理想，不代表它没有价值

更准确地说，它在当前系统里承担的是：

- 大结构稳定
- 低频锚定

而不是：

- 细边稳定
- 线条粗细稳定
- 留白边缘稳定

所以它的效果在热力图里表现为“局部有帮助，但不主导最终得分”。

### 3. 这张图已经足够支持一个明确工程判断

如果当前目标是“稳定优先”，那么现阶段不应把 `softedge_scale` 继续往大推。  
更合理的理解是：

- 先把 `softedge` 当成弱条件
- 先让 `depth` 负责大结构
- 先解决 `softedge` / `depth` 的时序归一化与动态调度

否则继续增大 `edge`，大概率只会进一步放大闪烁。

## 九、建议的下一步

### P0：先修 benchmark，避免图被实验流程污染

建议：

1. 每个参数组合开始前调用 `/api/reset/{user_id}`
2. 每个组合至少采样 3 次 `delta_avg_30s` 再取平均
3. 把 warmup 时间和正式采样时间分开记录
4. 除 `delta_avg_30s` 外，同时导出 `e_out` 与 `e_src`

否则后续图虽然还能看趋势，但很难用于精细调参。

### P0：参数层面的临时结论

如果目标是稳定优先，建议下一轮搜索先集中在：

- `softedge_scale: 0 ~ 0.2`
- `depth_scale: 0.4 ~ 1.0`

原因不是说 `softedge` 永远不能用，而是当前实现还没有把它稳定化，先把它作为弱条件更合理。

### P1：最值得优先改的推理逻辑

1. `softedge` 归一化从逐帧 `min/max` 改成更稳的 `裁剪 + EMA`
2. `depth` 归一化也改成 `分位数 + EMA`
3. 低运动场景显著降低 `min_noise_rate`
4. 为 `softedge_scale` 增加运动感知调度
5. 尝试把上一帧 `softedge` warp 到当前帧参与融合

### P1：如果还要继续用强 `softedge`

那就不应该只靠“复制 depth 分支权重 + 调 scale”。  
至少应考虑：

1. 单独为 `edge_flow_conv_in` 做轻量适配
2. 为 `softedge` 分支设计更稳的输入统计
3. 让 `softedge` 只在部分 timestep 或低运动区域更强介入

## 十、结语

这张图传达的不是“edge 无用”或“depth 无效”，而是：

- 当前 `softedge` 的实现方式，还没有把“边缘增强”变成“边缘稳定”
- 当前 `depth` 的作用范围，本来就更偏大结构，不足以单独压住水墨细闪

因此，图中“`edge` 越大越差、`depth` 改善有限”的现象，与当前代码逻辑是高度一致的，也与外部视频风格化、视频扩散、水墨视频迁移和视频深度研究的经验一致。

如果后续要把这条链路真正调成“稳定优先”的水墨模式，最先该做的不是继续盲目增大 `softedge_scale`，而是先把：

- 条件图时序统计
- 噪声注入策略
- `softedge` 的动态调度
- benchmark 的实验洁净度

这四件事收紧。
